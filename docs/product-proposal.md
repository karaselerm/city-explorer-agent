# Product Proposal — CityExplorer Agent (PoC)

## 1. Обоснование идеи

Проблема: пользователю сложно собрать прогулочный маршрут, который одновременно соответствует времени, дистанции, бюджету и тематике, а также имеет альтернативы при отказе источников данных.

Прикладная задача PoC:
- за 1 запрос получить выполнимый маршрут 2-4 часа;
- объяснить, почему выбраны именно эти точки;
- сгенерировать plan B, если часть данных недоступна или противоречива.

Почему это хороший проект под агентную систему:
- есть внешний KB (Overpass OSM);
- есть инструменты с side effects (экспорт файлов, опционально календарь);
- есть state/memory между сессиями;
- есть реальные failure modes (timeout, пустая выборка, конфликт constraints);
- есть требования по безопасности и наблюдаемости.

## 2. Цель и метрики успеха

### Цель

Построить PoC, который стабильно генерирует маршрут с соблюдением пользовательских ограничений и возвращает полезный результат даже при деградации внешних API.

### Продуктовые метрики

- `Route Acceptance Rate`: доля маршрутов, принятых пользователем без правок, цель >= 60%.
- `Median Edits Before Accept`: медиана числа правок, цель <= 1.
- `Plan B Usage`: доля кейсов, где пользователи использовали альтернативу, цель >= 20% (показывает практическую ценность fallback).

### Агентские метрики

- `Constraint Satisfaction`: доля ответов, где соблюдены лимиты по времени/дистанции и исключены avoid-категории, цель >= 90%.
- `Task Success Rate`: маршрут успешно собран и отрендерен, цель >= 95%.
- `Fallback Quality`: при срабатывании fallback результат все равно содержит >= 1 релевантную остановку, цель >= 95%.

### Технические и инфраструктурные метрики

- `p95 end-to-end latency`: цель <= 30 c.
- `p95 retriever latency`: цель <= 12 c.
- `Tool Error Rate`: цель <= 10%.
- `Availability (PoC runtime)`: >= 99% для локального запуска.
- `Cost guardrail`: без платных API по умолчанию.

## 3. Сценарии использования

### Основные

1. Туристический маршрут:
- "Москва, 3 часа, музей + парк + кофе, пешком, до 6 км".

2. Быстрый вечерний маршрут:
- "2 часа, тихо, недорого, без баров".

3. Тематический маршрут:
- "Культурный маршрут (музеи/исторические точки), без долгих переходов".

### Edge cases

1. Слишком узкие фильтры:
- мало кандидатов, система должна ослабить constraints и явно это объяснить.

2. Overpass timeout/rate limit:
- retriever переходит в fallback (кэш/локальные данные), не падает.

3. Противоречивые ограничения:
- "2 часа, 8 точек, максимум 2 км" -> маршрут усекается, пользователю возвращается предупреждение.

4. Неизвестный город:
- если нет bounds для live retrieval, система использует sample fallback и пишет warning.

5. Некорректный ввод:
- отрицательная длительность/дистанция -> валидация блокирует выполнение.

## 4. Ограничения

### Технические

- Ограничение шага retriever: timeout 12 c.
- Ограничение кандидатов: не более 100 POI на запрос.
- Ограничение stop budget: не более 6 остановок на маршрут.
- Execution budget оркестратора: ограниченный pipeline без бесконечных циклов.

### Операционные

- Бесплатные источники данных (Overpass + локальный датасет).
- Локальное хранение state в SQLite.
- Локальный запуск для команды 2-3 человека без сложной инфраструктуры.

## 5. Архитектурный набросок

Модули PoC:

1. `Orchestrator`
- порядок шагов, retry/fallback, stop conditions.

2. `Retriever`
- Overpass-запросы, нормализация POI, кэш, fallback dataset.

3. `Ranker`
- фильтрация must/avoid, scoring по бюджету/тихости/стилю.

4. `RouteBuilder`
- построение последовательности остановок и проверка constraints.

5. `Memory`
- профиль пользователя + история маршрутов (SQLite).

6. `Exporter`
- экспорт в `.md`, `.json`, `.ics` (c подтверждением).

```text
User Request
   ↓
Orchestrator Agent
   ↓ (tool call)
POI Retriever (Overpass) ──→ POI Candidates
   ↓
Filter & Rank ──→ Shortlist + Alternatives
   ↓ (tool call)
Route Builder ──→ Ordered Stops + Distance/ETA
   ↓
Renderer/Exporter ──→ Plan + Links + Files
```
   ↓
State/Memory updated (prefs, ratings, favorites)
=======
7. `Observability`
- structured logs, latency/fallback counters.

## 6. Data flow и зона ответственности LLM/Agent

LLM/Agent зона (в PoC допускается rule-based замена):
- orchestration decisions;
- объяснение результата;
- выбор fallback-ветки при деградации.

Детерминированные зоны:
- tool вызовы;
- фильтрация/валидация constraints;
- расчет дистанций и времени;
- экспорт и запись state.
