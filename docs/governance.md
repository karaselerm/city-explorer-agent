# Product Proposal — CityExplorer Agent (PoC)

## 1) Обоснование идеи (прикладная задача)

Пользователю нужно быстро получить **готовый план прогулки** (2–4 часа), который:
- логично организован (порядок остановок, переходы)
- учитывает ограничения (пешком/бюджет/тише/категории)
- имеет план B (если что-то недоступно/не понравилось)
- объясняет, почему предложено именно так

Решение — агентная система, которая:
- сама собирает кандидатов мест (инструменты),
- отфильтровывает/ранжирует (логика + LLM),
- строит маршрут (routing tool),
- хранит предпочтения и контекст (state/memory),
- оценивает качество и устойчивость (evals/monitoring).

## 2) Цель и метрики

### Цель PoC
Собрать MVP-агента, который по запросу пользователя строит корректный маршрут и выдаёт понятный план, минимизируя “пустые”/нерелевантные рекомендации.

### Метрики успеха

**Продуктовые:**
- User rating маршрута (1–5) по 3 критериям: релевантность / удобство / интересность
- Кол-во правок пользователя до принятия маршрута (чем меньше — тем лучше)
- Доля случаев, когда пользователь сохраняет маршрут (export/save rate)

**Агентские:**
- Task success rate: маршрут построен и соответствует ограничениям (время/дистанция/категории)
- Tool success rate: % успешных вызовов Overpass и routing без ручного вмешательства
- “Plan consistency”: маршрут не противоречит ограничениям (например, не предлагает бар при “без баров”)

**Технические:**
- p95 latency: время ответа (цель PoC: < 20–30 сек на “обычный” запрос)
- p95 tool latency + % timeouts
- Стоимость LLM (если применимо): лимит на 1 запрос (например, <$0.05–0.10)

## 3) Сценарии использования (включая edge-cases)

### Основные сценарии
1) **Классическая прогулка**
- “Сделай маршрут на 3 часа в районе X: кофе + музей + парк, пешком, недорого”.

2) **Тематический маршрут**
- “Архитектура модерна / стрит-арт / видовые точки”.

3) **Ограничения по времени и расстоянию**
- “2 часа, максимум 3 км, без музеев”.

4) **Сохранение и повтор**
- “Сохрани маршрут”, “Сделай похожее, но без точки №2”.

### Edge-cases (должны порождать ошибки и требовать failover)
- **Слишком мало POI** по фильтрам (“тихо + дешево + музей + район маленький”)
- **Overpass timeout / rate limit**
- **Плохие/неполные данные**: нет opening_hours, нет категории, странные теги
- **Сильная неоднозначность**: “центр” без указания города/страны (PoC может просить уточнение)
- **Противоречивые требования**: “мало ходить” + “пять точек в 2 часа”
- **Неожиданные форматы** (пользователь пишет смесь языков, опечатки)

## 4) Ограничения (технические и операционные)

### Технические (SLO / ограничения PoC)
- p95 ответ: < 30 сек (цель), при деградации — вернуть частичный план < 10 сек
- Устойчивость к сбоям инструментов: graceful fallback
- Ограничение по числу POI: например, максимум 100 кандидатов на запрос (чтобы не утонуть)

### Операционные
- Overpass может ограничивать частоту/долгие запросы → нужна локальная кэш-слой/дедупликация
- PoC без платных API (или с минимальными бесплатными лимитами)
- Хранение state/memory локально (SQLite/JSON), без персональных данных

## 5) Архитектурный набросок (модули и интеграции)

### Модули
1) **Orchestrator Agent**
- принимает запрос пользователя
- уточняет недостающие параметры (если критично)
- планирует шаги: retrieve → rank → route → render → export

2) **POI Retriever (Tool)**
- запрос к Overpass API
- нормализация результата (name, tags, coords, category, address)

3) **Ranker & Filter**
- правила + LLM (или правила + простая модель) для ранжирования
- фильтрация по категориям, предпочтениям, бюджету (proxy), “тишине” (proxy)

4) **Route Builder (Tool)**
- строит порядок точек и сегменты маршрута
- проверяет ограничение по времени/дистанции
- план B: альтернативы точек

5) **Renderer & Exporter (Tool)**
- вывод: маршрут + объяснение + ссылки
- экспорт: markdown/json, опционально `.ics` (с подтверждением)

6) **State & Memory**
- user_profile: предпочтения, “любимые”, история оценок
- session_state: цель, ограничения, выбранные точки, альтернативы
- storage: SQLite/JSON (PoC)

7) **Observability**
- логирование инструментов (успех/таймаут/latency)
- трассировка шагов агента (plan steps)
- метрики качества (success/fail, причины)

### Высокоуровневая схема

```text
User Request
   ↓
Orchestrator Agent
   ↓ (tool call)
POI Retriever (Overpass) ──→ POI Candidates
   ↓
Filter & Rank ──→ Shortlist + Alternatives
   ↓ (tool call)
Route Builder ──→ Ordered Stops + Distance/ETA
   ↓
Renderer/Exporter ──→ Plan + Links + Files
   ↓
State/Memory updated (prefs, ratings, favorites)

```
